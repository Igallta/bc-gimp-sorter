# MisakaChat 技术手册

> 最后更新：2026-07-20  
> 当前稳定版：MisakaChat **v2.10.15** / GimpSorter **v1.6.5**  
> 当前阶段：v2.10.15 半周黑箱测试；期间不继续扩张记忆查询逻辑。

本文是仓库内的长期技术记录，目的是避免开发战线拉长后遗失架构、决策、发布方式和后续方向。实现与本文冲突时，以当前运行代码为准，并应立即修正文档。

## 1. 项目目标

MisakaChat 是运行在 Bondage Club 聊天室中的御坂 Bot。它不是通用聊天机器人，核心目标是：

1. 以稳定、简短的御坂人设参与房间对话。
2. 记住人物、历史对话和少量长期事实。
3. 区分普通聊天、纯角色扮演和会真正改变 BC 状态的操作。
4. 在允许自然互动的同时，避免错误移动、错误穿戴和失败后的半成品状态。
5. 能在 BC、浏览器、插件和模型经常变化的环境中持续维护。

非目标：

- 不追求通用 Agent 平台或插件框架。
- 不把所有聊天都永久变成事实。
- 不让模型直接、不经验证地操作 BC 状态。
- 不依赖某个固定 BC 版本号。

## 2. 运行组成

```text
Tampermonkey
  └─ misaka-chat.user.js
       ├─ 固定 revision 加载 misaka-persona.js
       └─ 固定 revision 加载 misaka-chat.js
               │
               ├─ DeepSeek Chat Completions
               ├─ OpenRouter / OpenAI Embeddings
               ├─ localStorage：配置、人物档案、计数
               ├─ IndexedDB：语义记忆、提炼记忆
               └─ BC / ModSDK：消息、人物、道具、移动
```

### 文件职责

| 文件 | 职责 |
|---|---|
| `misaka-chat.user.js` | 域名匹配、账号限制、GM 能力桥接、版本与 runtime 固定 revision |
| `misaka-chat.js` | 对话、记忆、规划、真实操作、事务回滚、命令和初始化 |
| `misaka-persona.js` | 人设提示词、BC 资源中文名、目录与属性辅助 |
| `gimp-sorter.user.js` | GimpSorter loader |
| `gimp-sorter.js` | GIMP 娃娃编号识别与安静排序 |
| `bc-cn-translation.json` | BC Asset 的中文翻译数据 |

`backups/` 与 `backup-pre-skill-refactor/` 是历史快照，不参与线上运行。修改功能时不要误改这些文件。

## 3. 消息处理主链路

MisakaChat 收到房间消息后，大致经过：

```text
BC 消息
  ↓
过滤系统噪音、自己消息、冷却和触发条件
  ↓
更新 recentMessages / 人物档案 / 语义记忆
  ↓
独立规划器判定 intent
  ├─ chat       普通回复
  ├─ roleplay   只输出动作描写，不改变 BC 状态
  ├─ action     生成并执行真实 BC 操作
  └─ clarify    缺少必要信息时追问
  ↓
构造人设、房间名单、记忆和近期上下文
  ↓
主模型生成回复或操作命令
  ↓
记忆召回 / 操作执行 / 确定性验证
  ↓
格式清洗并发送到房间
```

### `roleplay` 与 `action`

这是 v2.10.9 后的重要边界：

- `咬 Rikka 一口` → `roleplay`，只发送 `*轻轻咬了一口*`。
- `给 Rikka 一份爆米花` → `action`，真正添加手持物。
- `把香肠递到 Rommy 嘴边` → 通常为 `roleplay`，不会把手持物错误装备到嘴部。

不要重新把咬、舔、拥抱、假装吃人等动作塞回真实道具执行层。

### 澄清状态

- 追问按发送者 MemberNumber 隔离。
- 默认承接窗口为 120 秒。
- 规划器只负责指出缺少什么，人设层负责把问题说得像御坂。

## 4. 真实操作与安全边界

真实操作支持移动、添加/删除/修改道具、颜色、表情、快照和束缚复制等。基本原则是：

1. **LLM 负责理解意图，代码负责验证结果。**
2. 操作前保存目标状态与房间位置。
3. 一轮操作可延迟人物同步，避免删除、添加、回滚包乱序。
4. 操作后检查客观后置条件。
5. 真正执行失败或违反明确限制时回滚。

v2.10.7 起，第二个 LLM 语义审查不再有权否决已经通过客观检查的操作；v2.10.9 已彻底移除该语义复核。仍保留的强制边界包括：

- 操作对象错误。
- 跨操作大类。
- `noMove`、`noAdd`、`noStack`、`preserveParts` 等明确限制。
- 未经规划的全解、快照恢复等高影响操作。
- 实际状态与目标不一致。

无害 ItemHandheld 不使用与拘束、移动相同的危险提示等级。

## 5. 当前记忆系统

当前实际运行的是四层记忆。

### 5.1 临时上下文 `recentMessages`

- 仅存在内存中。
- 最多保留最近 50 条。
- 上下文预算约 20,000 token。
- 页面刷新或脚本重载后消失。
- 用于理解当前连续对话和规划器最近 10 条语境。

### 5.2 人物档案 `localStorage.misaka_memory.profiles`

- 按 MemberNumber 区分人物。
- 保存名字、首次见面、最近互动、次数和备注。
- 本地最多保留 100 人。
- 每轮只选择房间内或保留分数最高的 20 人注入提示词。
- 这仍是“活跃人物档案”，不是完整的人际关系数据库。

### 5.3 语义记忆 `IndexedDB/misaka_chat/semantic_mem`

- 保存有意义的聊天原文、时间、说话者、MemberNumber、是否为御坂本人和 embedding。
- 单条文本最多 500 字，过短内容不存。
- 相似度高于 0.92 的近重复跳过。
- 最多 5,000 条；超限后按长度与时间价值淘汰低价值条目。
- 插件事件有时带有上下文意义，因此不进行粗暴的全量事件过滤。

### 5.4 提炼长期记忆 `IndexedDB/misaka_chat/refined_mem`

- 每累计 50 条有效外部聊天尝试提炼一次。
- 输入为最近 20 条可用语义记忆、相关人物档案和已有长期记忆。
- 每轮最多生成一条，最多保存 20 条。
- 只接受：关系、稳定偏好、边界、身份、持续状态。
- 排除玩笑、技术讨论、开发计划、一次性故障和未经本人确认的评价。
- 新候选与已有 `refined_mem` 做文字和向量去重，阈值 0.85。
- 所有长期记忆每轮全量注入，因此污染风险高于普通语义记忆。

### 5.5 当前语义查询链路（v2.10.15）

```text
主模型判断需要旧事细节
  ↓ 输出 [MEMSEARCH: 查询]
原始 cosine < 0.48 的候选拒绝
  ↓
返回最多 3 条候选 + 最强命中的前后各 1 条（5 分钟窗口）
  ↓
主模型基于候选起草回答
  ↓
记忆证据编辑器检查是否编造原因、语气、顺序和背景
  ↓
支持 / 修订 / 证据不足
```

时间只参与排序，是否相关由原始 cosine 决定，避免老但准确的记忆被时间衰减排除。

### 5.6 压缩记忆：当前未运行

历史上曾有：每 50 条消息生成不超过 80 字摘要，最多保留 5 段，并在每轮作为“之前对话摘要”注入。

该功能在 2026-07-07 的提交 `7f7c899`（大规模精简）中被一并删除。用户没有要求移除它，因此这是一次未经确认的功能回归，不应被误记为产品决策。

恢复压缩记忆前要先回答：它是否能补足刷新后的近期连续性，同时不与 `semantic_mem` 和 `refined_mem` 重复、污染或增加过多 token。不要直接照搬旧实现。

## 6. 存储与配置

### localStorage

| Key | 用途 |
|---|---|
| `misaka_memory` | 人物档案 |
| `misaka_msg_count` | 长期记忆提炼计数，刷新后继续 |
| `misaka_apikey` | 对话 API Key |
| `misaka_openrouter_key` | 首选 embedding Key |
| `misaka_openai_key` | 旧 OpenAI embedding 兜底 |
| `misaka_model` | 模型覆盖值 |
| `misaka_persona_extra` | 人设附加备注 |

API Key 只保存在本地浏览器，不得写入源码、提交、日志、文档或公开 issue。

### IndexedDB

数据库：`misaka_chat`，版本 2。

| Store | 内容 |
|---|---|
| `semantic_mem` | 原文、embedding、时间和说话者元数据 |
| `refined_mem` | 提炼事实、embedding 和时间 |

### 导入导出

- `/misaka export` 将数据库 JSON 放到 `window.__misakaExportData`。
- `/misaka import` 从该变量恢复。
- 大规模清理、迁移、改变 embedding 模型前必须先导出并校验备份。

## 7. 模型与外部服务

当前默认：

- 对话：DeepSeek Chat Completions，默认模型 `deepseek-v4-flash`。
- Embedding 首选：OpenRouter `qwen/qwen3-embedding-8b`。
- Embedding 旧兜底：OpenAI `text-embedding-3-large`，3,072 维。

注意：历史语义库主要为 3,072 维向量。切换 embedding 模型时必须检查维度兼容；不能把不同维度或语义空间的向量直接混合比较。必要时应使用备份后全量重新 embedding 的迁移流程。

## 8. Loader 与发布流程

MisakaChat loader 通过 `raw.githack.com` 加载固定 Git revision，而不是直接加载浮动 `master`。这是为避免 GitHub Pages/CDN 缓存导致 loader 与 runtime 版本不一致。

### 标准发布步骤

1. 确认工作区没有无关改动。
2. 修改 runtime，并执行定向回归。
3. 同步修改：
   - UserScript `@version`
   - loader `SCRIPT_VERSION`
   - runtime `SCRIPT_VERSION`
4. 先提交功能代码。
5. 将 `ASSET_REVISION` 指向包含该功能的 commit。
6. 再提交 release/pin commit。
7. 执行：

```bash
node --check misaka-chat.js
node --check misaka-persona.js
node --check misaka-chat.user.js
git diff --check
```

8. 推送后核对：
   - `origin/master` 与本地 HEAD 一致。
   - 固定 revision 能返回正确版本的两个 runtime 文件。
   - Tampermonkey 更新 loader 后刷新 BC，`/misaka status` 显示正确版本。

### BC 版本兼容

所有活跃用户脚本必须使用 `R*`，不得写死 `R129`、`R130`。升级 BC 后优先检查：

- userscript 是否注入。
- ModSDK hook 是否仍有效。
- BC Asset 名称、Group、Property 是否变化。
- `ChatRoomMessage`、人物移动与更新函数是否变化。

## 9. 调试与观测

### 内置状态

- `/misaka status`：版本、模型、key 来源、embedding、记忆数量和人物数。
- `window.__misakaDebugTrace`：最近 30 条内部 trace。
- 控制台前缀：`[MisakaChat]`、`[GimpSorter]`。

### 控制台拦截

OpenClaw 的页面 console 拦截器使用最多 1,000 条的环形缓存。MCP 休眠不会清掉页面缓存；刷新、跳转或标签页回收后需要重新注入。

### 当前测试方式

仓库目前没有正式自动化测试套件，主要依靠：

- `node --check` 语法检查。
- 隔离的函数/提示词定向样本。
- CDP 现场回归。
- 房间半周或一周黑箱测试。

这是明显的工程缺口。未来应优先把最稳定、最容易回归的纯函数测试固化，而不是继续增加提示词分支。

### 当前产品决策

- 插件事件有时具有真实上下文，不做粗暴全删。
- 操作审查保持宽松，客观错误和明确限制才阻止/回滚。
- 角色扮演与真实操作必须分开。
- 当前 v2.10.15 先做半周黑箱测试，不继续堆叠记忆分支。
- 未来简化记忆查询时，以 **v2.10.13** 作为“保留弱命中过滤、尚未引入复杂审核链”的参考基线。
- 不为了追求形式上的防幻觉，把御坂变成生硬的检索报告。

## 11. 已知问题与风险

### P0 / 近期关注

1. **记忆查询链路可能过度复杂。**  
   v2.10.15 比 v2.10.13 多一次证据审核，并允许有限的二次查询。收益要靠半周黑箱测试验证；若真实误答并未明显下降，应合并或删除分支。

2. **压缩记忆缺失。**  
   刷新后的近期连续性依赖语义召回，旧版 compaction 的缺失尚未正式解决。

3. **长期记忆只能追加，不能修订旧事实。**  
   关系或持续状态改变时，旧条目可能与新条目冲突。

### P1

4. **人物档案不是稳定关系库。** 保留分数仍可能淘汰旧人物并重置统计。
5. **语义库混有插件事件、乱码和短期闲聊。** 当前有意保留部分事件语义，后续应改进排序而非盲删。
6. **普通 chat 的动作/台词仍不是全局结构化输出。** `roleplay` 已确定性渲染，但普通回复仍可能偶发漏星号。
7. **缺少正式测试套件。** 大部分回归依赖现场和临时 harness。

### P2

8. 主文件超过 3,600 行，模块边界主要靠函数分区，维护成本继续上升。
9. 部分源码顶部的行号式模块注释会随改动过时，不应依赖其行号导航。
10. GimpSorter runtime 仍使用 GitHub Pages 浮动资源，与 MisakaChat 的固定 revision 发布方式不同。

## 12. 路线图

### 当前：半周黑箱测试

重点记录：

- 该查旧事时是否触发查询。
- 不该查时是否无意义查询。
- 弱命中时是否仍编造原因、语气或先后关系。
- 证据编辑器是否把正确、自然的推断误删。
- 多一次 LLM 调用带来的延迟、费用和失败率。
- 回答是否过度保守、生硬或频繁“不记得”。

不要只记成功案例。每个异常至少保留：原问题、候选片段、最终回答、期望行为和能否稳定复现。

### 黑箱测试后：记忆链简化评审

优先比较三个方案：

1. 保留 v2.10.15 全链路。
2. 回归 v2.10.13 式单轮召回，仅加强主提示词。
3. 保留证据审核，但取消二次查询或合并为一次结构化回答。

选择标准依次是：真实准确率、自然度、故障面、延迟/成本、代码复杂度。遵循奥卡姆剃刀：没有现场收益证明的机制不保留。

### 后续候选

- 设计新的近期压缩记忆，而非直接复活旧实现。
- 给长期记忆增加更新、替换和冲突解决。
- 把纯函数与存储逻辑拆出可测试模块。
- 固化 planner、roleplay、记忆召回和事务回滚的回归样本。
- 评估主文件模块化，但避免为了结构而结构。

## 13. 维护约定

每次形成长期有效的架构或产品决策时，至少更新本文以下一处：

- “重要版本与决策”
- “已知问题与风险”
- “路线图”

每次发布新版本时：

- 更新顶部稳定版。
- 在版本表增加一行关键变化。
- 删除已经解决的风险或标注解决版本。
- 若存储结构、key、模型或发布方式变化，同步更新对应章节。

不要把一次临时测试结论直接写成永久规则；先说明证据、适用版本和是否经过黑箱验证。
