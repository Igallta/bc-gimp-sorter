# MisakaChat 技术手册

> 最后更新：2026-07-20  
> 当前稳定版：MisakaChat **v2.10.17** / GimpSorter **v1.6.5**
> 当前阶段：v2.10.17 召回污染修复黑箱测试；v2.10.16 保留为架构基线。

本文是御坂项目的长期技术事实源。它不仅记录“代码现在怎么工作”，也记录“为什么会变成这样”：项目从一个整理 GIMP 娃娃站位的小脚本开始，逐步长成了一个能感知房间、理解自然语言、记住访客并真正改变 BC 状态的角色型 Bot。

Notion 中的《完整开发档案》和《系统架构与工程原则》是本文的重要来源。它们保留了很好的开发叙事和工程判断；本文在吸收其全部信息的同时，用当前源码和版本历史校正已经过时的状态。若本文与实现冲突，以当前运行代码为准，并应尽快修正文档。

## 1. 项目是什么

MisakaChat 运行在 Bondage Club 的 Gimp Dolls 房间中。她在设定上是流浪魔女咲（Misaki）的使魔与忠诚仆从，也是房间管理员和搬运工。

工程目标不是做一个会套话的聊天机器人，而是让御坂在 BC 世界里像一个真实角色：知道房间里有谁，看得见人物穿着，记得曾经发生的事，听得懂自然语言，并且说“做了”时就真的完成操作。

核心目标：

1. 以稳定、简短、有温度的御坂人设参与房间对话。
2. 感知人物、关系、穿着、道具、站位、时间和进出记录。
3. 记住熟人、历史对话和少量具有长期价值的事实。
4. 区分普通聊天、纯角色扮演和真正改变 BC 状态的操作。
5. 在允许自然互动的同时，避免错人物、错部位、错道具和失败后的半成品状态。
6. 能在 BC、浏览器、插件和模型持续变化的环境中维护和发布。

非目标：

- 不做通用 Agent 平台或插件框架。
- 不把每段闲聊、玩笑和操作请求都永久变成事实。
- 不让模型绕过确定性检查直接决定 BC 的最终状态。
- 不依赖固定的 BC 版本目录。
- 不在公共聊天室暴露 API、Prompt、模型和内部执行细节。

## 2. 从 GimpSorter 到 MisakaChat：完整演进

截至 2026-07-16，仓库已经累计约 366 个提交。主要开发集中在 7 月 1 日以后；许多看似复杂的机制，都来自真实 BC 房间中的一次具体失败。

### 阶段 0：先成为可靠的搬运工（7 月 1 日）

项目最初只有 GimpSorter。它识别 `GIMP XXX` 名字中的编号，并按编号整理房间站位。

- 从 v1.3 起，反复比较 `MoveLeft`、`MoveRight`、批量移动和逐步移动的稳定性。
- 最终保留周期检查、移动后等待服务器同步、再根据实时顺序继续的实现。
- 加入 Tampermonkey loader、御坂账号白名单、日志开关和 GitHub 托管。

这一阶段形成了最早也最重要的原则：**BC 客户端状态会异步变化，不能只在本地算完就假定成功；必须重新观察真实状态。**

### 阶段 1：让御坂能听见并自然回应（7 月 2 日）

MisakaChat v1 接入 LLM 后，御坂第一次真正能参与房间聊天。

- 触发方式从必须 `@` 改为符合 BC 习惯的名字触发：提到 Misaka、御坂或御搬即可回应。
- 解决 BC 消息字段、CSP、请求方式和可见发送路径，最终通过页面上下文桥接请求，并使用 BC 自己的输入框和发送函数发言。
- 处理重复 Hook、旧实例残留、重复回复、并发锁、冷却、超时和重新注入。
- 建立第一版稳定人格：安静、简短、略带冷幽默和傲娇；中文为主；不承认自己是 AI 或脚本。
- 把房间名单、昵称、MemberNumber、主人/恋人、在线描述、穿着、束缚数量、颜色和 GIMP 身份交给模型。
- 接入 BCE 本地档案和房间进出记录，让她能查询不在房间中的人物资料与上次出现时间。
- 将动作与台词拆开发送，清理名字前缀、引号、固定口癖和机械化格式。

### 阶段 2：从“会说”到“会做”（7 月 3–4 日）

御坂随后从聊天角色变成真正能操作 BC 状态的角色。

- 从硬编码道具表迁移到运行时遍历 Asset，曾能识别约 1,353 个 Item 资产。
- 加入 BC 官方中文翻译、别名、同义词和身体部位映射。
- 建立隐藏操作指令：
  - `MOVE`：水平移动、移动到某人旁边或房间边缘；
  - `ITEMADD / ITEMDEL`：添加、移除或释放道具；
  - `ITEMSET`：设置 Typed、Modular、Vibrating 等属性与样式；
  - `ITEMCOLOR`：整体或指定 layer 改色；
  - `SNAPSHOT / COPY`：保存、恢复和复制束缚状态；
  - 后续增加 `EMOTE`、`BCEQUERY` 和 `MEMSEARCH`。
- 解决权限 API 静默失败、Appearance 直接修改、同名 Asset 多 Group、数组删除变异、属性广播和颜色层长度等实际兼容问题。

这一阶段确立了“不能只演”的硬规则：**操作请求必须真的产生可执行指令并改变状态；没有执行就不能用动作描写或一句‘好了’冒充完成。**

同一时期还加入了人物档案、语义记忆、提炼长期事实、10 分钟 idle 和最早的上下文压缩。

### 阶段 3：更像御坂，也更诚实（7 月 4–8 日）

早期模型很容易把 Prompt、thinking、名字前缀、HTML 实体和游戏事件带到公屏。随后进行了多轮人格与事实边界整理：

- 修复 thinking 泄漏、空回复、动作格式、时间戳模仿和自动消息污染。
- 让御坂知道当前日期与时间，并加入自然 idle 和新人欢迎；欢迎可关闭。
- 房间名单没有的人不能编；没有足够记忆时要承认不记得。
- 不能把操作请求推断成长期偏好，也不能把御坂自己的话误记成他人的事实。
- 记忆从 localStorage 迁入 IndexedDB，加入 embedding、相似去重、时间衰减、智能遗忘和重建 embedding。
- 形成三线召回：概括事实每轮注入、具体旧对话按需向量搜索、人物档案单独维护。
- 接入 BCE 查询、完整穿着、进出记录、表情气泡、实时房间状态和按需道具目录。
- API Key 从仓库移到浏览器本地，避免公开提交和 GitHub 扫描。

### 阶段 4：自然语言规划与事务安全（7 月 9–14 日）

关键词分流无法稳定理解“放我出来”“关进笼子”“把手绑住”等长尾表达，因此系统改为独立自然语言规划器。

- 规划器先区分聊天、操作和澄清，再给出目标人物、部位、精确 Asset、允许操作和限制。
- 明确 `noMove / noAdd / noStack / preserveParts / replaceExisting` 等约束。
- 修复代词：`我/给我` 指发送者；`你/御坂/你自己` 指御坂；称呼“御坂”本身不能被误当成操作宾语。
- 澄清状态按发送者保存，其他人插话不会打断；短答继承原目标与限制，默认两分钟过期。
- “进入设备”被定义为 ItemDevices 状态操作，不能用水平 `MOVE` 冒充进入笼子或家具。
- 复合命令保持原始顺序，替换时先删旧、再加新、最后设样式。
- 加入状态感知、精确部位、精确 Asset、命名姿势和 Modular 属性验证。
- 命名姿势必须落到 BC 的真实属性上；例如“驷马缚”不能只靠文字声称完成，而要验证实际使用了对应的 Hogtie 样式。
- 操作前备份 Appearance 与站位，延迟统一同步；任一子步骤失败或最终状态不符时回滚。
- 通过后置条件检查道具存在/消失、颜色、Typed/Modular/Vibrating 属性，以及禁止移动、新增、叠加和保留部位等约束。

典型回归是：他人插话后，咲只补充“御坂，狗窝”，系统仍能承接上一轮请求，把目标人物的 `PetBed` 正确替换为 `LowCage`。

这一轮多发送者澄清与 loader 修复的重要提交包括 `e6155c9`、`5ce06b4`；具体提交顺序和完整变更仍以 Git 历史为准。

### 阶段 5：v2.10.7 与第一轮真实黑箱测试（7 月 14–15 日）

v2.10.6 已能完成复杂设备替换、`noStack` 和两端状态检查。随后开始一周真实黑箱测试，集中记录误操作、漏操作、格式和记忆问题。

最先暴露的两个问题：

1. idle 偶发生成 `*动作*台词*`，整句被 BC 当作 Action。
2. 客观状态已正确，第二个 LLM 语义验收却因措辞差异将操作驳回并回滚。

v2.10.7 将 idle 动作与台词确定性拆分，并把语义复核降为只读意见。真正的执行失败、后置条件失败和明确限制仍会回滚。修复与 loader 固定提交分别为 `42eb58a`、`821b30b`。

### 阶段 6：角色扮演、宽松审查与跨版本发布（v2.10.8–10）

- v2.10.8 保留 thinking，仅把 idle 输出预算从 80 提高到 1,024；拒绝用残句关键词拦截正常语言。
- v2.10.9 增加 `roleplay`，把咬、舔、躲藏、假装吃人等可见动作与真实 BC 操作分开。
- 规划器开始读取最近 10 条对话，改善“也要”“那个”“手一份腿两份”等指代。
- 澄清文案交回人格层，避免客服式追问。
- 移除第二个 LLM 操作审查；无害手持食物不再使用拘束级风险提示。
- 补齐爆米花、鸡腿、香肠等手持食物目录和复制/转移/角色扮演语义。
- v2.10.10 将 Userscript 从固定 `R129` 改为 `R*`，覆盖 BC 未来版本与主要镜像域名。

### 阶段 7：长期记忆卫生（v2.10.11–12）

长期运行后，20 条 `refined_mem` 被玩笑、技术讨论、残句和重复内容占满。v2.10.12 完成清理与结构升级：

- 清理前导出并校验完整数据库备份。
- 20 条污染长期记忆清理为 3 条有依据的稳定事实。
- 提炼只接受关系、偏好、边界、身份和持续状态。
- 去重改为与 `refined_mem` 自身比较，而不是误查 `semantic_mem`。
- 人物档案容量从 20 提升到 100，每轮仍只注入最相关的 20 人。
- 标准 Activity/Action 不再增加互动次数和长期提炼计数。

插件事件有时本身带有重要上下文，因此系统没有粗暴删除全部事件文本。

### 阶段 8：真实记忆召回与复杂度反思（v2.10.13–15）

- v2.10.13 拒绝原始 cosine 低于 0.48 的弱命中，加入日期、相似度和强命中邻近上下文。
- v2.10.14 强化“询问过去时先查，再承认不记得”，修复该查却不查的问题。
- v2.10.15 增加证据编辑器和有限二次查询，防止把候选片段拼成不存在的原因、顺序和背景。

隔离测试显示触发准确率和事实准确率明显提高，但调用链与代码复杂度也上升。当前决定是先做半周黑箱测试，期间不再扩张；之后以 v2.10.13 为简化参考基线，遵循奥卡姆剃刀评估哪些环节真正有现场收益。

### 阶段 9：单路记忆 RAG（v2.10.16）

v2.10.15 黑箱测试确认：证据编辑器能改善“查到以后别瞎编”，但主回复模型仍会随机漏掉本应执行的查询。直接回到 v2.10.13 并只加强提示词，重复测试中触发率明显退化。因此 v2.10.16 改为：

- 由每轮本来就会运行的结构化规划器输出 `memorySearch` 和 `memoryEntities`；
- 用“人物名 + 用户原话”查询，不让模型重写事件，从而保留主客体关系；
- 同时查询 `semantic_mem` 与 `refined_mem`，继续使用 0.48 门槛、Top 3 和最强命中邻近上下文；
- 由一个专用记忆回答器直接输出 `supported / conflict / insufficient` 和最终回复；
- 无命中或证据不足时确定性收口，不再让聊天模型根据问题本身补故事；
- 删除自由草稿、独立证据编辑器、二次改写查询、第三轮生成和搜索循环状态机。

为抵消规划器自身的随机漏判，v2.10.16 只保留一条窄而确定的保险：句子同时包含明确过去时间标记和疑问表达时，强制进入记忆查询。它不使用宽泛关键词猜测普通聊天意图。

### 阶段 10：阻断现场召回污染（v2.10.17）

v2.10.16 的隔离测试通过，但现场测试连续返回 `insufficient`。根因不是规划器或向量库失效，而是消息处理顺序：当前用户问题会先写入 `semantic_mem`，随后立刻参与同一问题的检索；旧代码还把“用户问题 → 御坂回复”再次合成为一条语义记忆。当前问题和历史兜底对拥有极高字面相似度，因而会挤出真正的旧对话。

v2.10.17 保持单路 RAG 架构不变，只修正召回边界：

- 召回时排除最近两分钟的语义记录，近期内容继续由聊天上下文负责；
- 排除明确写给御坂的历史提问和旧的“问题 → 回复”合成记录；
- 停止继续生成这种合成语义记忆，只分别保留真实聊天原文；
- 查询只补用户原文中缺失的第三方人物，不再加入说话者或御坂自己；
- 对御坂自己的回复轻微降权，优先使用他人的原始发言；
- 记忆回答器使用确定性参数，并对“是不是/吗”确认题只保留足以回答问题的首个事实分句，防止追加无依据评价。

既有合成记录没有删除；它们只是在召回阶段被忽略，因此这次修复不改写历史数据库。

## 3. 当前运行组成

```text
Tampermonkey
  └─ misaka-chat.user.js
       ├─ 固定 revision 加载 misaka-persona.js
       └─ 固定 revision 加载 misaka-chat.js
               │
               ├─ DeepSeek Chat Completions
               ├─ OpenRouter / OpenAI Embeddings
               ├─ localStorage：配置、人物档案、计数
               ├─ IndexedDB：语义记忆、提炼记忆
               └─ BC / ModSDK：消息、人物、道具、移动
```

### 文件职责

| 文件 | 职责 |
|---|---|
| `misaka-chat.user.js` | 域名匹配、账号限制、GM 能力桥接、版本与固定 runtime revision |
| `misaka-chat.js` | 对话、记忆、规划、真实操作、事务回滚、命令和初始化 |
| `misaka-persona.js` | 人格提示词、BC 资源中文名、目录与属性辅助 |
| `gimp-sorter.user.js` | GimpSorter loader |
| `gimp-sorter.js` | GIMP 娃娃编号识别与安静排序 |
| `bc-cn-translation.json` | BC Asset 中文翻译数据 |

`backups/` 与 `backup-pre-skill-refactor/` 是历史快照，不参与线上运行。

### 当前运行参数

- 对话模型：`deepseek-v4-flash`，最大输出 8,192 tokens。
- 短期上下文：最多 50 条、约 20,000 tokens。
- 全局冷却 3 秒；单用户冷却 5 秒；拟人化发送延迟 800ms。
- 一轮复杂操作可能包含规划、主回复、纠错和验收，hard timeout 为 180 秒。
- idle：10 分钟无人说话触发，每分钟检查一次。
- 澄清承接窗口：120 秒。

## 4. 能力流水线与当前能力

御坂不是“聊天模型加一堆命令”，而是一条以实时 BC 状态为权威、以自然语言计划为中介、以事务执行和确定性验收收口的角色型 Bot 流水线：

1. **感知**：读取发送者、房间成员、关系、在线描述、Appearance、道具、时间与进出记录。
2. **理解**：结合短期上下文、人物档案、提炼事实和按需召回，判断是聊天、角色扮演、真实操作还是澄清。
3. **规划**：把自然语言转为目标人物、部位、Asset、允许动作和限制。
4. **生成**：按御坂人格输出台词与隐藏操作指令。
5. **执行**：按原顺序执行移动、道具、样式、颜色、快照和复制。
6. **验收**：重新读取 Appearance 与站位，验证后置条件。
7. **回滚**：任一确定性条件失败时恢复操作前状态，并给出简短、诚实的角色化反馈。

当前能力包括：

- 名字触发、上下文聊天、时间感知和 CN/EN 为主的角色化回复。
- 房间成员、身份关系、Bio、完整穿着、颜色、实时 Appearance 和束缚状态读取。
- BCE 离线档案查询、进出记录、自动 idle 和可选新人欢迎。
- 移动人物、添加/移除/释放道具、改色、设置属性与样式、设备替换、快照、复制束缚和表情气泡。
- 对第三方人物执行操作；真正高影响操作保留确定性边界。
- 自然语言规划、按发送者澄清、计划外指令过滤、事务执行、后置验收和自动回滚。
- `/misaka` 启停、模型、Key、状态、记忆导入导出、人格备注和清理命令。

## 5. 消息处理主链路

```text
BC 消息
  ↓
过滤自己消息、系统噪音、冷却和触发条件
  ↓
更新 recentMessages / 人物档案 / 语义记忆
  ↓
独立规划器判定 intent
  ├─ chat       普通回复
  ├─ roleplay   可见动作，不改变 BC 状态
  ├─ action     生成并执行真实 BC 操作
  └─ clarify    缺少必要信息时追问
  ↓
构造人格、房间名单、记忆和近期上下文
  ↓
主模型生成回复或隐藏操作命令
  ↓
记忆召回 / 操作执行 / 确定性验证
  ↓
格式清洗并发送到房间
```

### `roleplay` 与 `action`

- `咬 Rikka 一口` → `roleplay`，只发送动作描写。
- `给 Rikka 一份爆米花` → `action`，真正添加手持物。
- `把香肠递到 Rommy 嘴边` → 通常为 `roleplay`，不会把手持物错误装备到嘴部。

不要把咬、舔、拥抱、躲藏和假装吃人重新塞回道具执行层。

### 澄清状态

- 按发送者 MemberNumber 隔离。
- 默认承接窗口 120 秒。
- 别人插话不会清掉原请求。
- 规划器只说明缺少什么，人格层负责把问题说得像御坂。

## 6. 真实操作与安全边界

基本原则是：**LLM 负责理解意图，代码负责验证结果。**

1. 操作前保存目标 Appearance 与房间位置。
2. 事务中可延迟同步，避免删除、添加、失败恢复的网络包乱序。
3. 按模型原始命令顺序执行，不擅自重排。
4. 操作后重新读取状态并检查后置条件。
5. 真正失败或违反明确限制时回滚。

v2.10.7 起，第二个 LLM 语义审查无权否决已经通过的客观事实；v2.10.9 已彻底移除该复核。

仍保留的强制边界：

- 操作对象错误或跨操作大类。
- `noMove`、`noAdd`、`noStack`、`preserveParts` 等明确限制。
- 未经规划的全解、快照恢复等高影响操作。
- Asset、颜色、Typed/Modular/Vibrating 属性或最终位置与目标不符。

无害 ItemHandheld 不使用与拘束和移动相同的危险提示等级。内部技术诊断只进入本地日志，公屏只发送简短、诚实、符合人设的结果。

## 7. 当前记忆系统

### 7.1 临时上下文 `recentMessages`

- 只存在内存，最多 50 条。
- 预算约 20,000 tokens。
- 页面刷新或脚本重载后消失。
- 用于当前连续对话；规划器读取最近 10 条语境。

### 7.2 人物档案 `localStorage.misaka_memory.profiles`

- 按 MemberNumber 区分人物。
- 保存名字、首次见面、最近互动、次数和备注。
- 本地最多 100 人；每轮只注入房间内或保留分数最高的 20 人。
- 它仍更接近“活跃熟人缓存”，不是完整关系数据库。

历史上限曾是 20 人，导致熟人被淘汰后重新认识、`firstSeen` 和互动次数重置；v2.10.12 已缓解，但没有从根本上把档案变成永久关系库。

### 7.3 语义记忆 `IndexedDB/misaka_chat/semantic_mem`

- 保存聊天原文、时间、说话者、MemberNumber、是否为御坂和 embedding。
- 单条最多 500 字，过短内容不存。
- cosine 高于 0.92 的近重复跳过。
- 最多 5,000 条；超限后按长度与时间价值淘汰低价值项。
- 插件事件有时具有上下文意义，因此不进行粗暴的全量过滤。

2026-07-14 的历史审计曾有 594 条；后续增长到三千余条。数值属于运行状态，不应固定写进 Notion 首页，实时数量以 `/misaka status` 和数据库为准。

### 7.4 提炼长期记忆 `IndexedDB/misaka_chat/refined_mem`

- 每累计 50 条有效外部聊天尝试提炼一次。
- 输入为最近 20 条语义记忆、相关人物档案和已有长期记忆。
- 每轮最多一条，最多保存 20 条。
- 只接受关系、稳定偏好、边界、身份和持续状态。
- 排除玩笑、技术讨论、开发计划、一次性故障和未经本人确认的评价。
- 与已有 `refined_mem` 做文字和向量去重，阈值 0.85。
- 所有长期事实每轮全量注入，所以污染风险高于普通语义记忆。

2026-07-14 曾将 20 条污染记录清理为 6 条；这些后来被 FIFO 淘汰。v2.10.12 再次备份并清理为 3 条可靠事实，同时修复提炼分类和去重对象。

### 7.5 当前语义查询链路（v2.10.17）

```text
结构化规划器判断是否需要旧事检索
  ↓ 需要时使用“人物名 + 用户原话”
同时检索 semantic_mem 与 refined_mem
  ↓ 排除近期问题、历史提问与合成对话
  ↓ 0.48 门槛、Top 3、最强命中邻近上下文
专用记忆回答器一次生成 supported / conflict / insufficient 最终回复
  ↓
无结果或证据不足时确定性收口
```

时间只影响排序，相关性门槛使用原始 cosine，避免老但准确的记忆被时间衰减误杀。候选不一定属于同一事件，不能为了说得完整而强行拼接。

v2.10.15 的回退链路是：

```text
主模型输出 MEMSEARCH
  ↓ 检索并起草回答
独立证据编辑器审核
  ↓ 必要时有限二次查询
最终回复
```

v2.10.17 沿用 v2.10.16 的单路调用，不经过普通人格回复，也不允许再次搜索；生成与证据约束被合并为一次模型调用。变化只发生在召回边界和确认题的确定性收口。

### 7.6 压缩记忆：当前未运行

旧版曾每 50 条消息生成不超过 80 字的摘要，最多保存 5 段，并作为“之前对话摘要”每轮注入。它能补足刷新后的近期连续性，也曾作为长期提炼输入。

该功能在 2026-07-07 的 `7f7c899` 大规模精简中被一并删除。用户从未要求移除，因此这是未经确认的功能回归，不是产品决策。

若恢复，应先证明它能补足近期连续性，而不会与 `semantic_mem`、`refined_mem` 重复、污染或显著增加 token；不要直接复活旧实现。

## 8. 存储、模型与外部服务

### localStorage

| Key | 用途 |
|---|---|
| `misaka_memory` | 人物档案 |
| `misaka_msg_count` | 长期提炼计数 |
| `misaka_apikey` | 对话 API Key |
| `misaka_openrouter_key` | 首选 embedding Key |
| `misaka_openai_key` | 旧 OpenAI embedding 兜底 |
| `misaka_model` | 模型覆盖值 |
| `misaka_persona_extra` | 人格附加备注 |

Key 只保存在本地浏览器，不得写入源码、提交、日志、文档或公开 issue。

### IndexedDB

数据库：`misaka_chat`，版本 2。

| Store | 内容 |
|---|---|
| `semantic_mem` | 原文、embedding、时间和说话者元数据 |
| `refined_mem` | 提炼事实、embedding 和时间 |

### 模型

- 对话：DeepSeek Chat Completions，默认 `deepseek-v4-flash`。
- Embedding 首选：OpenRouter `qwen/qwen3-embedding-8b`。
- 旧兜底：OpenAI `text-embedding-3-large`，3,072 维。

历史语义库主要是 3,072 维向量。切换 embedding 模型必须检查维度和语义空间兼容，必要时备份后全量重建。

### 导入导出

- `/misaka export` 将数据库 JSON 放到 `window.__misakaExportData`。
- `/misaka import` 从该变量恢复。
- 大规模清理、迁移或更换 embedding 前必须先导出并验证备份。

## 9. 人格、公开信息与 Bio

核心身份已经稳定：Misaki 的忠诚使魔与仆从，Gimp Dolls 房间管理员兼搬运工。

人格原则：

- 温柔、有点傲娇和冷幽默，但不靠固定口癖表演。
- 被叫到会回应，不冷淡、不像客服。
- 能调侃眼前行为，但不替真人判断人格、善恶、动机或关系立场。
- 动作和情绪保持连续；被绑、被夸或被调侃后，不应下一句立刻完全遗忘。
- 不承认自己是 AI、脚本或模型；技术问题以角色化方式回避。

公开 Bio 已设计为自然描述她作为使魔、管理员、熟人记忆、人物移动和束缚/穿着操作的能力，不暴露内部架构。早期 Notion 档案中“新版 Bio 尚未上线”是 v2.10.7 时的历史状态，不应再作为当前技术事实。

## 10. Loader、域名与发布流程

MisakaChat 通过 `raw.githack.com` 加载固定 Git revision，而不是浮动 `master`，避免 CDN 缓存制造 loader 与 runtime 不一致。

### 标准发布步骤

1. 确认没有无关改动。
2. 修改 runtime，并执行定向回归。
3. 同步 UserScript `@version`、loader `SCRIPT_VERSION` 和 runtime `SCRIPT_VERSION`。
4. 提交功能代码。
5. 将 `ASSET_REVISION` 指向包含功能的 commit。
6. 再提交 release/pin commit。
7. 运行：

```bash
node --check misaka-chat.js
node --check misaka-persona.js
node --check misaka-chat.user.js
git diff --check
```

8. 推送后核对本地与 `origin/master`、固定 revision、Tampermonkey loader 和 `/misaka status`。

### BC 版本兼容

活跃 Userscript 必须使用 `R*`，不能写死 `R129` 或 `R130`。覆盖主要 BC 镜像与 localhost。BC 升级后优先检查：

- Userscript 是否注入。
- ModSDK Hook 是否有效。
- Asset 名称、Group 和 Property 是否变化。
- `ChatRoomMessage`、人物移动和同步函数是否变化。

普通 Chrome 与 CDP Chrome 的 Tampermonkey 数据、固定 runtime 和缓存曾多次制造“安装页已更新，实际运行仍是旧版”的错觉。发布确认必须以页面真实版本为准。

## 11. 调试、黑箱测试与回归

### 内置观测

- `/misaka status`：版本、loader、模型、Key 来源、embedding、记忆数量和人物数。
- `window.__misakaDebugTrace`：最近 30 条内部 trace。
- 控制台前缀：`[MisakaChat]`、`[GimpSorter]`。

OpenClaw 的页面 console 拦截器使用最多 1,000 条的环形缓存。MCP 休眠不会清掉页面缓存；刷新、跳转或标签页回收后要重新注入。

### 黑箱测试最小记录

每个异常至少保存：

- 时间、御坂版本、发送者和相关人物。
- 原始请求与前两轮上下文。
- 当时 Appearance、道具、站位和明确限制。
- 御坂回复、计划、指令、实际状态与是否回滚。
- 期望回复、期望操作和应通过/失败的后置条件。
- 分类、严重度、能否复现、修复版本、提交和验收证据。

复盘顺序：

1. 先处理改变他人状态、污染长期记忆或稳定复现的问题。
2. 再处理漏操作、错目标、错部位、错道具和错误回滚。
3. 最后处理纯措辞、重复 idle 和轻微人格偏差。
4. 修复前保留原始记录，修复后在同一案例补验收证据。

### 当前测试缺口

仓库尚无正式自动化套件，主要依靠：

- `node --check`。
- 隔离函数和 Prompt 样本。
- CDP 现场回归。
- 半周或一周真实房间黑箱测试。

应优先把稳定的 planner、roleplay、记忆召回、命令解析、后置条件和回滚样本固化，而不是继续增加 Prompt 分支。

## 12. 当前状态、技术债与风险

### 当前产品判断

工程的核心已经不是继续堆能力，而是把真实房间异常转成可重放测试，让每次修复有证据、每次发布可验证、每条长期事实可追溯。

### 近期风险

1. **单路记忆 RAG 仍需真实房间验证。** 隔离测试表现良好，但规划器触发、冲突识别和自然度仍需黑箱测试证明。
2. **压缩记忆缺失。** 刷新后的近期连续性主要依赖语义召回。
3. **长期事实只能追加。** 关系和持续状态改变时，旧事实不会自动修订。
4. **语义库仍含插件事件、乱码和短期闲聊。** 部分事件有上下文价值，应改善排序而非盲删。
5. **人物档案不是永久关系库。** 高价值旧人物仍可能被淘汰并重置统计。
6. **普通 `chat` 不是全局结构化动作/台词输出。** `roleplay` 已确定性渲染，普通回复仍可能偶发格式错误。
7. **规划仍依赖 LLM。** 模组道具、锁、权限、复杂 layer 和多人复合操作仍有长尾风险。
8. **复杂链路有延迟和 API 成本。** 规划、回复、召回、审核、纠错和执行会扩大故障面。
9. **部分状态仅在内存。** 束缚快照刷新后会丢失。
10. **主文件超过 3,600 行。** 模块边界主要靠函数分区，顶部行号注释会随修改过时。
11. **发布链可能制造版本错觉。** loader、revision、Tampermonkey profile 和缓存必须同时核对。
12. **GimpSorter runtime 仍使用浮动资源。** 与 MisakaChat 的固定 revision 方式不同。

## 13. 路线图

### 当前：v2.10.17 召回污染修复黑箱测试

重点观察：

- 现场聊天写入顺序下是否仍会由当前问题或旧兜底对抢占候选。
- 规划器与窄规则组合后的查询触发率和误查率。
- `supported / conflict / insufficient` 三类回答是否稳定。
- 人物名与用户原话组合查询是否避免主客体颠倒。
- 确认题的首事实收口是否过度裁剪有效信息。
- 回答是否变得过度保守、生硬或频繁“不记得”。

### 后续：记忆链简化评审

比较：

1. 保留 v2.10.15 全链路。
2. 回到 v2.10.13 式单轮召回，只加强主提示词。
3. 使用 v2.10.16 单路 RAG：规划器触发、原话检索、一次结构化回答。

依次比较真实准确率、自然度、故障面、延迟/成本和代码复杂度。没有现场收益证明的机制不保留。

### 后续方向

- 建立可离线重放的回归框架，覆盖代词、插话后澄清、设备替换、命名姿势、部位、颜色 layer、锁、多人操作和失败回滚。
- 收紧规划器 JSON schema；确定性代码负责常见执行语法，LLM 只理解目标。
- 将服务器确认纳入关键事务，扩大模组、锁、复杂属性和多人操作的后置检查。
- 设计新的近期压缩记忆，而非直接复活旧版。
- 给长期事实增加来源、置信度、更新、替换、冲突和过期机制。
- 人物偏好与关系事实绑定 MemberNumber，避免模糊的“用户”归因到错误对象。
- 提供单条记忆查看、删除和纠正；测试模式默认不污染长期事实；若清理语义库，只精确删除开发测试、失败生成、内部验收和完全重复，不粗暴删除具有上下文意义的插件事件。
- 根据说话者改善 CN/EN/JP，继续优化 idle、新人欢迎和 Bio 一致性。
- 持久化需要跨刷新保存的快照，并增加过期和手动清理。
- 记录调用耗时、失败类型、回滚次数和 token/API 使用，仅用于本地诊断。
- 提供一键导出最近 debug trace、计划、隐藏指令、执行结果和验收结果，减少现场排障时手工抄控制台。
- 评估模块化，但避免为了结构而结构。

御坂服务器常驻、聊天室身份验证桥接和人物资料同步属于独立的 Gimp Dolls Chatroom 项目，只在明确交界点与本项目关联，不应混入 MisakaChat 的技术路线图。

## 14. 验收指标

- **操作成功率**：明确可执行请求中最终状态正确的比例。
- **错误操作率**：错人物、错部位、错道具或违反限制的比例。
- **漏操作率**：口头答应却未执行，或应执行却被当成聊天的比例。
- **安全回滚率**：失败后是否完整恢复 Appearance 与站位。
- **澄清质量**：是否问对缺失信息，短答后能否继承上下文。
- **记忆准确率**：长期事实是否有证据，是否出现对象归因和测试污染。
- **召回忠实度**：是否只陈述候选支持的内容，证据不足时能否收口。
- **对话自然度**：前缀、动作格式、thinking、机械复读和无关 idle 的发生率。
- **发布一致性**：UserScript、loader、runtime 与固定 revision 是否一致。

## 15. 工程原则与维护约定

### 核心原则

- **实时 BC 状态是最高权威。** 当前 Appearance 与房间顺序高于历史消息和模型猜测。
- **客观检查优先。** 模糊的 LLM 意见不能推翻已经验证的状态事实。
- **说做了，就必须真的做了。** 没有执行和最终确认，不能用台词冒充完成。
- **不会就问，不记得就先查。** 信息不足时澄清；召回无证据时承认不记得。
- **复合操作必须是事务。** 半完成通常比明确失败更危险。
- **角色扮演与真实操作分开。** 玩笑和动作描写不能误改 BC 状态。
- **先收集真实案例，再修改系统。** 稳定复现比继续堆 Prompt 更有价值。
- **技术能力服务于角色。** 玩家看到的是咲的使魔御坂，不是内部模块和诊断信息。
- **遵循奥卡姆剃刀。** 没有现场收益证明的额外模型调用和分支不保留。

### 文档维护

每次形成长期有效的架构或产品决策，至少更新本文的演进史、风险或路线图之一。每次发布：

- 更新顶部稳定版。
- 在演进史或关键版本中写清真正改变行为的部分。
- 删除已解决风险，或标明解决版本。
- 存储结构、Key、模型或发布方式变化时同步对应章节。

不要把临时测试结论直接写成永久规则；应说明证据、适用版本和是否经过真实黑箱验证。
